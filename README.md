# Secure Checkout Sandbox

Flask-based UPI checkout sandbox with:

- session-protected checkout access
- backend-generated order IDs
- protected transaction receiver endpoint
- UTR-based payment verification
- Supabase persistence for checkout and received-payment rows
- Socket.IO updates for live status changes

## Run

Install dependencies:

```bash
pip install -r requirements.txt
```

Set environment variables:

- `FLASK_SECRET_KEY`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_TABLE=checkout_orders`
- `PAYMENT_RECEIVED_TABLE=payment_received`
- `TRANSACTION_INGEST_API_KEY`
- `SOCKET_CORS_ALLOWED_ORIGINS`
- `API_CORS_ALLOWED_ORIGINS`

Run the app:

```bash
python app.py
```

## Database

Apply [`supabase_schema.sql`](./supabase_schema.sql) in Supabase to create:

- `checkout_orders`
- `payment_received`

## Core Flow

1. Client creates an order with `POST /api/orders`.
2. Backend generates `order_id`, `txn_id`, and a one-time checkout token.
3. Client opens `/checkout/<order_id>?token=...`.
4. Checkout page collects customer details and asks for UTR.
5. Receiver server sends raw transaction description to `POST /api/payments/receive`.
6. App parses UTR from the transaction description and stores it in `payment_received`.
7. User enters UTR on checkout page and clicks verify.
8. App checks whether that UTR exists in `payment_received`.
9. App compares payment amount with order amount.
10. If matched, the payment is marked verified and linked to exactly one order.

## Security Model

- Checkout access is protected by a signed Flask session.
- The initial checkout URL uses a one-time token.
- The receiver endpoint requires `X-API-Key`.
- UTRs are unique in `payment_received`.
- Verified UTRs cannot be reused for another order.
- Order verification uses backend-side lookup and amount comparison.
- Client-facing error messages are generic and do not expose backend table names.

## Endpoints

### 1. Create Order

`POST /api/orders`

Creates a checkout order from only:

- `upi_id`
- `merchant_name`
- `amount`
- `description`

#### Request headers

```http
Content-Type: application/json
```

#### Request body

```json
{
  "upi_id": "merchant@upi",
  "merchant_name": "Acme Corporation Pvt. Ltd.",
  "amount": 49999,
  "description": "Demo checkout session"
}
```

#### Response

```json
{
  "message": "Order created",
  "order": {
    "order_id": "ORD-20260505-ABC123",
    "txn_id": "TXN-20260505091500-1A2B",
    "amount": 49999.0,
    "currency": "INR",
    "merchant_name": "Acme Corporation Pvt. Ltd.",
    "upi_id": "merchant@upi",
    "note": "Demo checkout session",
    "status": "pending",
    "verification_status": "pending",
    "status_message": "Awaiting payment confirmation"
  },
  "checkout_url": "/checkout/ORD-20260505-ABC123?token=...",
  "checkout_token": "..."
}
```

#### Notes

- `order_id` is generated by the backend.
- `checkout_token` is one-time access proof for the checkout page.
- Customer details are not sent at order creation.

### 2. Open Checkout

`GET /checkout/<order_id>?token=...`

Opens the secure checkout page after token validation.

#### Query parameters

- `token`: one-time checkout token returned by `POST /api/orders`

#### Behavior

- validates the token
- stores a Flask session
- renders the checkout wizard

#### Example

```text
/checkout/ORD-20260505-ABC123?token=eyJ...
```

### 3. Fetch Order

`GET /api/orders/<order_id>`

Returns the current public order state for the active checkout session.

#### Headers

```http
Cookie: session=...
```

#### Response

```json
{
  "order_id": "ORD-20260505-ABC123",
  "txn_id": "TXN-20260505091500-1A2B",
  "amount": 49999.0,
  "merchant_name": "Acme Corporation Pvt. Ltd.",
  "upi_id": "merchant@upi",
  "status": "pending",
  "verification_status": "pending",
  "status_message": "Awaiting payment confirmation"
}
```

### 4. Receive Transaction Description

`POST /api/payments/receive`

This is the receiver-server webhook. It accepts the raw transaction description message, parses the UTR, and stores the row in `payment_received`.

#### Required headers

```http
Content-Type: application/json
X-API-Key: <TRANSACTION_INGEST_API_KEY>
```

`X-Webhook-Secret` is also accepted as an alias, but `X-API-Key` is the preferred header.

#### Request body

```json
{
  "transaction_description": "UPI payment received a/c XX5549 INR 250.00 on 23-04-26 16:33:53 Ref No UTR1234567890"
}
```

#### Optional aliases

The endpoint also accepts:

- `description`
- `message`
- `text`

#### Response

```json
{
  "message": "Transaction description stored",
  "parsed_utr": "UTR1234567890",
  "payment_received": {
    "payment_id": "TXN-20260505091510-1A2B",
    "utr": "UTR1234567890",
    "amount": 250.0,
    "currency": "INR",
    "verification_status": "pending",
    "status_message": "Transaction received and stored in payment_received."
  }
}
```

#### What gets stored

The row is saved in `payment_received` with:

- `payment_id`
- `utr`
- `amount`
- `currency`
- `transaction_description`
- `parsed_transaction`
- `verification_status`
- `status_message`
- timestamps

---

### 5. Verify UTR From Checkout

`POST /api/orders/<order_id>/verify-utr`

This is called from the checkout page after the user enters their UTR.

#### Required headers

```http
Content-Type: application/json
Cookie: session=...
```

#### Request body

```json
{
  "utr": "UTR1234567890",
  "customer_name": "Aditya Chaudhari",
  "customer_email": "aditya@example.com",
  "customer_mobile": "9876543210",
  "reference_note": "Order payment"
}
```

#### Verification rules

- The UTR must exist in `payment_received`.
- The payment amount must match the checkout order amount.
- The UTR must not already be linked to another order.
- If the same order is already verified with the same UTR, the request is treated as idempotent.

#### Success response

```json
{
  "message": "UTR verification processed",
  "matched": true,
  "already_verified": false
}
```

If the order was already verified with the same UTR:

```json
{
  "message": "Order already verified",
  "matched": true,
  "already_verified": true
}
```

#### Failure responses

- `404 Unidentified UTR`
- `409 UTR already used`
- `422 Verification failed`

The client receives generic messages only.

### 6. Simulate Status

`POST /api/orders/<order_id>/simulate`

#### Request body

```json
  "message": "Simulated verification event"

```

## Socket.IO Events

Socket.IO is used for live order status updates.

### Client events

#### `subscribe_order`

Sent after the checkout page connects.

```json
{
  "order_id": "ORD-20260505-ABC123"
}
```

#### `ping_order`

Optional status refresh event.

```json
{
  "order_id": "ORD-20260505-ABC123"
}
```

### Server events

#### `verification_update`

Broadcast when order status changes.

#### `verification_error`

Emitted for unauthorized or invalid checkout sessions.

## Example Receiver Server Snippet

Python `requests` example:

```python
import requests

response = requests.post(
    "http://127.0.0.1:5000/api/payments/receive",
    headers={
        "Content-Type": "application/json",
        "X-API-Key": "change-me-too",
    },
    json={
        "transaction_description": (
            "UPI payment received a/c XX5549 INR 49999.00 "
            "on 23-04-26 16:33:53 Ref No UTR1234567890"
        )
    },
    timeout=10,
)

print(response.status_code)
print(response.json())
```

Equivalent cURL:

```bash
curl -X POST "http://127.0.0.1:5000/api/payments/receive" ^
  -H "Content-Type: application/json" ^
  -H "X-API-Key: change-me-too" ^
  -d "{\"transaction_description\":\"UPI payment received a/c XX5549 INR 49999.00 on 23-04-26 16:33:53 Ref No UTR1234567890\"}"
```
